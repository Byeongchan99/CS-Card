## 질문
유니티에서 Instantiate와 Destroy는 각각 내부적으로 무슨 일을 하길래 비싼가? 특히 Destroy를 호출하면 메모리가 그 즉시 반환되는가?

## 핵심 답변
Instantiate는 프리팹 원본 데이터를 읽어 게임오브젝트·컴포넌트 구성을 복제하고, 그에 대응하는 네이티브(C++) 객체를 엔진 메모리에 만들고, C#에서 다룰 수 있게 C# 래퍼 객체를 힙에 만들어 둘을 매핑한다. 이 복제·초기화·래퍼 생성 과정에서 관리 힙 할당이 발생해 나중에 GC가 수거할 가비지가 된다. Destroy는 즉시 메모리를 반환하지 않는다. 네이티브 객체의 실제 소멸은 해당 프레임 처리 끝에 이뤄지고, C# 래퍼는 그것과 별개로 참조가 다 끊긴 뒤 GC가 수거한다.

## 정리

### 내부 동작
Instantiate가 동반하는 작업:
- 프리팹 원본 데이터를 읽어 필요한 게임오브젝트·컴포넌트 구성 복제
- 대응하는 네이티브(C++) 객체를 엔진 메모리에 생성
- C# 스크립트가 다룰 수 있게 C# 래퍼 객체 생성 + 네이티브와 매핑
- 이 과정의 복제·초기화·래퍼 생성에서 관리 힙 할당 발생 → 가비지 누적

Destroy가 동반하는 작업:
- 네이티브 객체의 실제 소멸은 즉시가 아니라 해당 프레임 처리 끝에 수행
- C# 래퍼는 Destroy와 무관하게, 도달 가능한 참조가 모두 끊긴 뒤 GC가 수거

### 왜 그런가
유니티의 GameObject/Component는 실체가 C++ 쪽에 있고, C#에서 만지는 변수는 그 실체를 가리키는 얇은 래퍼다. 이 이원 구조 때문에 하나의 논리적 객체가 "네이티브"와 "C# 래퍼" 두 부분으로 나뉘고, 생성 시엔 둘 다 만들어야 하고 소멸 시엔 각각 다른 주체(엔진 vs GC)가 다른 시점에 처리한다.

### 흔한 오해/함정
"Destroy를 부르면 그 자리에서 다 사라진다"는 오해. 프레임 끝에 처리되는 것은 네이티브 소멸이고, C# 래퍼는 참조가 남아있으면 계속 살아있다. 이 둘의 죽음은 주체도 시점도 다르다.
