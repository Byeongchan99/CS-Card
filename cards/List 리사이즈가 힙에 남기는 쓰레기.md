## 질문
List&lt;T&gt;의 리사이즈가 일어나면 교체된 기존 내부 배열은 어떻게 되는가? 이 동작이 유니티에서 매 프레임 new List&lt;T&gt;()를 만들거나 캐파시티 지정 없이 Add로 채우는 코드가 문제가 되는 이유와 어떻게 연결되는가?

## 핵심 답변
내부 배열은 힙에 있고, 리사이즈로 교체된 옛 배열은 참조가 끊겨 GC 수거 대상, 즉 쓰레기가 된다. 매 프레임 리스트를 새로 만들면 리스트 객체와 내부 배열이 프레임마다 쓰레기로 쌓이고, 캐파시티 미지정 Add 반복은 중간 크기의 배열들을 연쇄적으로 만들었다 버린다. 이렇게 누적된 쓰레기가 GC 수거를 촉발하며, 유니티에서는 그 수거가 프레임 히치(GC 스파이크)로 나타난다.

## 정리
### 내부 동작
```
Add 진행:   [4] → [8] → [16] → [32] → ...
                 ↑버려짐  ↑버려짐  ↑버려짐
```
리사이즈는 "새 배열 할당 → 복사 → 참조 교체"이므로 CPU 비용(복사)과 GC 비용(버려진 배열)이라는 두 청구서를 동시에 발행한다. 시간 복잡도 분석은 앞의 것만 보고, 게임 성능 문제는 주로 뒤의 것에서 터진다.

### 실무(게임 개발)에서 생기는 문제와 해결
핫 패스(Update 등)에서의 규율은 "GC를 빠르게"가 아니라 "쓰레기를 만들지 않기"다. 리스트는 한 번 만들어 재사용하고 프레임 시작에 Clear하며, 예상 크기를 알면 생성자에서 캐파시티를 지정한다. 반복 생성·파괴되는 게임 오브젝트는 [[오브젝트 풀링]]으로 같은 원리를 적용한다.
