## 질문
유니티 객체의 오버로딩된 == 는 순수 C# 참조 비교보다 왜 비싼가? Update()에서 매 프레임 수십 개의 유니티 오브젝트에 null 체크를 도는 코드가 있다면 이를 어떻게 완화할 수 있는가?

## 핵심 답변
오버로딩된 == 는 비교할 때마다 네이티브 쪽 상태(매핑 유효성)를 확인하므로, 단순 포인터 비교인 순수 C# 참조 비교보다 비싸다. Update()에서 매 프레임 이 체크를 수십 개씩 돌리면 그 비용이 누적된다. 완화책은 두 갈래다. 하나는 생존 여부를 순수 C# 값(bool 등)으로 캐싱해 매 프레임 네이티브 확인을 피하는 것, 다른 하나는 애초에 파괴를 하지 않는 오브젝트 풀링을 써서 네이티브가 죽을 일 자체를 없애 fake null 체크가 필요 없게 만드는 것이다.

## 정리

### 왜 그런가
순수 C#의 == 는 참조 두 개를 비교하는 값싼 연산이다. 반면 유니티의 오버로딩 == 는 fake null을 정확히 판정하려고 매번 네이티브 상태까지 조회한다. 이 추가 조회가 개별로는 작아도 매 프레임 × 다수 객체로 곱해지면 무시 못 할 비용이 된다.

### 실무(게임 개발)에서 생기는 문제와 해결
- 캐싱: 자주 참조하는 유니티 객체의 생존/유효 여부를 순수 C# 상태로 들고 있어 매 프레임 == 호출을 줄인다. 단, 네이티브가 코드 모르게 죽는 경로에서 캐시가 거짓말할 수 있으므로 갱신 시점 설계가 중요하다.
- 풀링: 파괴 대신 활성/비활성만 토글하면 네이티브가 죽지 않아 fake null 자체가 안 생기고, 상태를 순수 C# 값(활성 리스트/비활성 큐)으로 관리할 수 있다.

### 트레이드오프
캐싱은 값싼 비교를 얻는 대신 캐시 무효화(언제 갱신할지) 문제를 떠안는다. 풀링은 이 문제를 구조적으로 소거하지만 상시 메모리 점유와 반납 관리 비용이 든다.

### 흔한 오해/함정
"null 체크는 공짜"라는 가정. 순수 C#에서는 거의 공짜지만 UnityEngine.Object에서는 아니다. 핫패스의 반복 null 체크는 프로파일링 대상이 될 수 있다.
