## 질문
유니티가 사용하는 가비지 컬렉터는 무엇이며, 닷넷(CoreCLR) 표준 GC와 비교했을 때 구조적으로 어떤 차이가 있는가?

## 핵심 답변
유니티는 Boehm-Demers-Weiser(BDW) GC를 사용하고, 프레임 드랍을 막기 위해 마크 작업을 여러 프레임에 쪼개는 Incremental GC가 적용되어 있다. 닷넷 GC는 세대별 수집으로 단명 객체를 싸게 처리하고 압축(compaction)으로 단편화를 해소하지만, 유니티의 Boehm GC는 세대 구분 없이 전체 힙을 마크하고 압축을 지원하지 않아 단편화가 누적되기 쉽다. 이 모든 차이의 뿌리는 Boehm이 타입 정보 없이 비트 패턴만 보고 포인터를 추측하는 conservative 방식이라는 데 있다.

## 정리

### 개념
- Boehm-Demers-Weiser GC: 언어·런타임에 독립적으로 얹을 수 있는 범용 conservative 수집기. 유니티가 이식성을 위해 채택했다.
- 유니티 스크립팅은 IL2CPP로 C# → C++ 네이티브 변환되며, 이 네이티브 세계에는 CoreCLR의 세대별·압축형 GC가 딸려오지 않는다. 그래서 별도의 범용 GC가 필요했다.

### 왜 그런가
- 닷넷 GC는 mark-compact + 세대(generation) 구조. 대부분의 객체가 금방 죽는다는 경험칙(generational hypothesis)에 기대 Gen0만 자주 수거하면 싸게 끝난다. 살아남은 객체를 힙 한쪽으로 몰아 압축해 단편화도 없앤다.
- 유니티 Boehm GC는 세대가 없어 매번 전체 힙을 마크한다. Incremental GC는 이 마크를 시간축으로 쪼갤 뿐, 세대로 나누는 게 아니다.

### 트레이드오프
- 닷넷: 정밀(precise) GC라 압축·세대가 가능하지만, 런타임이 타입 정보를 정확히 추적해야 해 언어 종속적이다.
- 유니티/Boehm: conservative라 어떤 언어에도 얹기 쉽지만(이식성 최우선), 압축 불가·세대 없음·단편화라는 대가를 치른다.

### 흔한 오해/함정
- "유니티 GC도 세대별이라 단명 객체는 싸다" → 틀림. 전체 힙 마크다.
- "Incremental GC가 켜지면 GC 부담이 사라진다" → 아니다. 총 작업량은 같고 프레임당 분산될 뿐이다. 오히려 write barrier 오버헤드가 추가된다.
