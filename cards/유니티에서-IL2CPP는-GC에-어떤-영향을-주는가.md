## 질문
스크립팅 백엔드를 Mono에서 IL2CPP로 바꾸면 매니지드 힙과 GC는 어떻게 달라지는가?

## 핵심 답변
달라지지 않는다. 유니티는 Mono와 IL2CPP 양쪽 모두에 Boehm GC를 쓰며, 매니지드 힙도 그대로 존재한다. IL2CPP는 IL을 C++로 번역하는 AOT 컴파일러이고, 함께 배포되는 런타임 라이브러리(libil2cpp)가 GC를 제공한다. 생성된 C++ 코드에는 할당·해제를 위한 BDW GC 호출이 삽입되므로, C#의 "참조가 있으면 살아있다"는 의미론이 번역 후에도 유지된다. 바뀌는 것은 컴파일 방식과 그로 인한 AOT 제약이지 메모리 모델이 아니다.

## 정리

### 왜 그런가
C#은 참조가 남아있는 한 객체가 살아있어야 하고 개발자가 delete를 호출하지 않는다. 번역된 C++가 순수한 new를 쓴다면 해제 주체가 없어 전부 누수가 된다. 언어를 바꿨다고 메모리 모델까지 바뀌면 같은 프로그램이 아니므로, C# 의미론을 유지하려면 번역 결과에도 GC가 반드시 존재해야 한다. 그래서 IL2CPP의 C++ 출력은 평범한 new가 아니라 GC 할당자를 호출한다.

### 내부 동작
il2cpp.exe가 IL을 C++ 소스로 번역하고 플랫폼별 C++ 컴파일러가 네이티브 바이너리를 만든다. 런타임 라이브러리 libil2cpp가 GC, 스레드·파일 접근, 내부 호출 구현을 제공하며 여기에 Boehm-Demers-Weiser GC가 포함된다. 네이티브 코드가 매니지드 객체를 붙들어야 할 때는 GCHandle을 만들어 그 객체를 GC 루트로 등록하고, 그러면 해당 객체와 그것이 참조하는 것들이 회수되지 않는다.

### 왜 컨서버티브인가
IL2CPP가 컨서버티브 GC를 쓰는 이유는 네이티브 C++ 코드와의 통합 용이성과 전 플랫폼 이식성이다. 정밀 GC는 어느 메모리 영역에 힙 포인터가 있는지 런타임이 정확히 알아야 성립한다. 그런데 번역된 C++를 최적화하는 주체는 플랫폼별 C++ 컴파일러이고, 그 컴파일러가 참조를 어느 레지스터에 두는지 스택 어디에 흘리는지를 유니티가 통제할 수 없다. 따라서 스택·CPU 레지스터·정적 메모리를 전부 훑어 힙 주소로 해석 가능한 값을 잠재 포인터로 보는 방식이 사실상 강제된다.

이 선택의 대가는 Mono 백엔드에서와 동일하다. non-generational, non-moving이라 compaction이 불가능하고, 오식별로 죽은 객체가 살아남으며, blacklisting이 위험 페이지를 할당에서 배제해 단편화를 남긴다. 관련 카드의 내용이 IL2CPP 빌드에도 그대로 적용된다.

### 실무(게임 개발)에서 생기는 문제와 해결
IL2CPP 전환으로 GC 부담이 줄어든다는 기대는 틀렸다. 프레임당 할당을 0에 가깝게 유지하기, 오브젝트 풀링, 컬렉션 캐파시티 사전 확보 같은 처방은 백엔드와 무관하게 유효하다. IL2CPP 빌드의 크래시 콜스택에 GC_mark_from 등 bdwgc 프레임이 찍히는 것이 같은 수집기라는 직접 증거다.
GarbageCollector API로 플레이어에서 GC를 완전히 비활성화할 수 있으나, 비활성 구간에서는 초과 할당이 곧바로 메모리 증가로 이어지므로 로딩처럼 수명이 짧고 통제된 구간에 한정해 쓴다.

### 흔한 오해/함정
- "C++로 변환되니 GC가 사라진다"가 대표적 오해다. 컴파일 방식만 바뀐다.
- 래퍼/네이티브 이원 구조로 설명하려는 시도도 어긋난다. 그 구조는 UnityEngine.Object를 상속한 엔진 객체에만 해당하며, 네이티브 짝이 없는 평범한 C# 클래스도 IL2CPP는 동일하게 번역하고 동일하게 GC 힙에 올린다.
