## 질문
Destroy 직후 C# 래퍼 객체는 힙에 멀쩡히 살아있는데도 go == null 비교가 true를 반환한다. 실제 C# 객체가 존재하는데 어떻게 null과 같다고 나오는가?

## 핵심 답변
UnityEngine.Object가 == 연산자를 오버로드하기 때문이다. 비교 시점에 래퍼가 가리키는 네이티브 객체가 아직 유효한지(내부 매핑이 살아있는지)를 검사하고, 네이티브가 이미 파괴됐으면 true를 반환한다. 그래서 C# 래퍼는 힙에 존재하지만 "유니티 관점의 null"로 취급된다. 이렇게 실제로는 살아있는 참조가 null처럼 보이는 상태를 fake null이라고 부른다.

## 정리

### 왜 그런가
유니티 객체는 네이티브(C++) 실체 + C# 래퍼의 이원 구조라, 네이티브만 죽고 래퍼는 살아있는 어긋난 상태가 생긴다. 유니티는 이 어긋남을 사용자에게 감추기 위해 ==(및 bool 변환)를 오버로드해, 네이티브가 죽은 래퍼를 null처럼 보이게 만든다.

### 내부 동작
- 순수 C#의 == 는 참조(포인터) 비교뿐이지만,
- UnityEngine.Object의 == 는 비교 시점에 네이티브 상태 유효성까지 확인한다.
- 네이티브가 죽었으면 래퍼가 힙에 있어도 == null 이 true.

### 흔한 오해/함정
"go == null 이 true면 C# 객체도 힙에서 사라졌다"는 오해. 아니다. 래퍼는 살아있고 네이티브만 죽은 상태다. 이 차이를 모르면 왜 null 체크가 통과했는데도 GC 메모리가 안 줄어드는지 등에서 혼란이 생긴다.
