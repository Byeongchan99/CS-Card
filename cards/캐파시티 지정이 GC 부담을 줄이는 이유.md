## 질문
캐파시티 지정 없이 빈 List&lt;T&gt;에 1000개를 Add하면 힙에는 무엇이 몇 개나 만들어졌다 버려지는가? 이를 근거로 캐파시티 지정이 복사 비용 절감과는 별개로 GC 부담을 줄이는 이유를 설명하라.

## 핵심 답변
.NET의 List&lt;T&gt;는 빈 상태에서 첫 Add 시 캐파시티 4로 시작해 두 배씩 커지므로, 4, 8, 16, ..., 512 크기의 중간 배열 8개가 만들어졌다 버려지고 1024짜리만 살아남는다. 버려진 배열들의 총 크기(약 1020)는 최종 배열과 맞먹으므로, 캐파시티 미지정은 필요한 메모리의 약 두 배를 힙에 할당했다 버리는 셈이다. new List&lt;T&gt;(1000)으로 시작하면 중간 배열이 아예 생기지 않아 복사 비용(CPU)과 쓰레기 발생량(GC) 두 청구서가 동시에 사라진다.

## 정리
### 내부 동작
```
캐파시티:  4 → 8 → 16 → 32 → 64 → 128 → 256 → 512 → 1024
버려짐:    4   8   16   32   64   128   256   512        (8개, 합 ≈ 1020)
```
등비수열의 성질상 버려진 총량 ≈ 최종 배열 크기. 즉 쓰레기 발생량이 두 배가 된다.

### 실무(게임 개발)에서 생기는 문제와 해결
크기를 예측할 수 있는 컬렉션(적 최대 수, 발사체 최대 수 등)은 반드시 캐파시티를 지정해 생성한다. Boehm GC는 압축을 하지 않으므로 다양한 크기의 중간 배열이 반복 생성·폐기되면 힙 파편화도 함께 악화된다.

### 흔한 오해/함정
캐파시티 지정의 이득을 "복사 비용 절감"으로만 답하면 절반이다. 게임 성능 관점에서 더 중요한 이득은 쓰레기 발생 자체를 없애 GC 수거 빈도와 스파이크를 줄이는 것이다.
